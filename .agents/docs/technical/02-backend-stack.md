# 后端技术栈

## 1. 目标与职责

后端负责：

- 房间生命周期（等待、开始、结束）
- 状态机推进（夜晚/白天/结算）
- 规则裁定（胜负检测、技能结算、投票与平票）
- 实时消息广播与私发（角色信息、阶段提示、结果同步）

## 2. 推荐栈

| 类别 | 方案 | 用途 |
|------|------|------|
| 运行时 | `Node.js >= 24` | Termux 中运行服务（推荐 24.x LTS） |
| 语言 | `TypeScript` | 类型约束与可维护性 |
| HTTP 框架 | `Fastify` | 处理 join/start 等接口 |
| 实时通信 | `@fastify/websocket` + `ws` | 推送阶段变化与接收玩家操作 |
| 数据存储 | 内存对象（Map/State 容器） | 单局会话数据 |
| 入参校验 | `zod` | 校验 HTTP/WS 事件载荷，保护状态机入口 |
| 日志 | `pino`（Fastify 内置） | 结构化日志与问题追踪 |

## 3. 模块划分建议

```text
server/
  src/
    app/                 # 服务启动与插件注册
    game/
      engine/            # 状态机与规则裁定
      models/            # 玩家、角色、投票、死亡记录等模型
      presets/           # 6-12 人标准预设
    transport/
      http/              # /api/join 等 HTTP 接口
      ws/                # WS 连接、事件路由、广播私发
    shared/
      types/             # 共享类型
      constants/         # 事件名、状态枚举
```

## 4. WebSocket 选型结论

- 标准模式首版不使用 Socket.IO
- 理由：
  - 局域网短链路场景下，原生 WS 能满足需求
  - 事件模型简单，自己维护事件信封更透明
  - 依赖更少、排障更直接，适合手机 Termux 环境
- 预留升级条件：
  - 未来要支持跨公网联机、复杂房间管理、强依赖自动重连/ACK 机制时，再评估 Socket.IO

## 5. 协议边界（与产品文档一致）

- `POST /api/join`：加入游戏并分配会话身份
- `game:start`：主机发起开始，服务端验证人数与预设
- `night:*` / `day:*`：玩家在对应阶段提交动作
- 服务端广播 `game:phase_change`、`game:vote_result`、`game:game_over` 等事件

## 6. 关键实现约束

- 服务端权威：客户端任何输入都必须做“阶段+身份+存活状态”校验
- 幂等处理：重复提交不应破坏状态一致性
- 单线程安全：所有状态变更通过统一调度入口执行
- 对局内不可改配：预设在开局锁定

## 7. 第一版可选增强（建议）

1. 连接层心跳与超时踢出（减少僵尸连接）
2. 操作日志（便于复盘与排查）
3. 规则单测（夜晚结算、胜负检测、平票重投）
4. 断线倒计时统一调度器
5. `node:test` + `c8` 覆盖率门槛（先从引擎核心规则开始）
